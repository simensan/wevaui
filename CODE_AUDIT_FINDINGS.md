## Memory hazards â€” pool imbalance

Audit basis: `PaintCommandPool`, `PaintListPool`, `PaintConverterPools`, `LayoutPools` (`BoxPool`, `LayoutScratch`), `CssValuePool`, `CascadePools`. Every Rent/Allocate site was diffed against its corresponding Return/Release. No exception path skips a return on the explicit pool surfaces â€” the only allocator that uses an explicit try/finally (BackgroundResolver.argsListPool, `BackgroundResolver.cs:501-544`) is correctly fenced. The findings below cover non-obvious imbalances rather than fictitious leaks.

- [ ] **ML1** â€” `LayoutScratch.RentFlexItemsView` / `RentLineView` / `RentBoxView` / `RentBlockBoxView` / `RentGridItemPropsView` / `RentIntView` / `RentGridPartialView` / `RentGridSizingItemView`. `Packages/com.wevaui/Runtime/Layout/LayoutPools.cs:239-363`. The Rent / Return helpers run inside flex / grid layout entry points; if `Layout()` throws after Rent but before the matching Return (e.g. a `GridTrackParser.ParseException` propagating from a sub-pass, or an `InvalidOperationException` from a length context), the rented `List<T>` is **dropped from the pool** until the LayoutEngine recycles. The pool then allocates a fresh List on the next Rent. Impact: bounded â€” each pool tops out at 4-8 lists (per the stack guard at `:243-247`, `:271-274`, etc.), so a single exception loses at most one list per pool kind. Single per-event impact: ~1 KB transient. Long-term: zero growth (pool refills naturally). Flag retained because the entry points do NOT use try/finally; a callers' refactor that introduces a new throw site would slowly drain the pools without obvious symptoms.
- [ ] **ML2** â€” `BoxPool.Allocate*` returns a fresh box (`!DisablePooling && freeList.Count > 0 ? Pop : new`) and unconditionally adds it to `allocated` (`Packages/com.wevaui/Runtime/Layout/LayoutPools.cs:55-125`). If `LayoutEngine.Layout()` throws between `BeginPass()` and `EndPass(survivedRoot)`, `allocated` keeps growing across subsequent Layout calls â€” `BeginPass()` calls `allocated.Clear()` on entry, which means an aborted pass's boxes are silently dropped from the pool's reach (they become GC garbage rather than reused). Impact: bounded by GC; not a leak per se but a "pool starves on exception" pattern. The free-lists themselves are unbounded (`Stack<BlockBox>(64)` initial cap but no max), so a long-running session with many exception-aborted layouts produces fresh `new BlockBox()` allocations forever. Comment at `:19-23` claims "Pool size is unbounded â€” there is no soft cap" deliberately.
- [ ] **ML3** â€” `CssValuePool.RentLength/RentNumber/RentPercentage` outside an active scope (`activeScopeDepth == 0`) returns `new CssLength(...)` / etc. and does NOT track the instance in any live list. `Packages/com.wevaui/Runtime/Css/Values/CssValuePool.cs:65-92, :115-138, :140-163`. Documented at `:17-19` ("Pooling is OPT-IN: when no CssValuePoolScope is active, Rent* allocates fresh instances without recording them"). Not a leak because the caller owns the reference â€” flagged for completeness because the asymmetry between "scope active â†’ tracked, will be returned" vs "scope inactive â†’ leaked-to-GC" is easy to miss when reading the code. The 4096-cap fallback path inside an active scope (`:25-27`) has the SAME behaviour: instances allocated above the cap are NOT tracked and become GC garbage on scope exit. Bounded but not pool-reclaimable.
- [ ] **ML4** â€” `PaintCommandPool.ReturnOne` short-circuits on `Pop*` commands and `Unknown` (`Packages/com.wevaui/Runtime/Paint/Conversion/PaintCommandPool.cs:288-290`). Pop commands are singletons (`PaintCommandSingletons.cs:127-134`) so this is correct. **BUT** if a future PaintCommand kind is added and the `switch` isn't updated, the new command silently leaks (falls through, never returned to any pool, never GC-collected until the PaintList itself is dropped). Defensive: add a `default: Debug.Assert(false, "unhandled paint command kind")`.
- [ ] **ML5** â€” `PaintListPool.Shared` (`Packages/com.wevaui/Runtime/Paint/PaintListPool.cs:73`) is a **process-static singleton** with a 8-list cap. UIDocument's per-frame Convert rents from Shared and Returns at the end of EmitPaint â€” fine in steady state. **However**, multiple UIDocument instances all draw from / Return to the same Shared pool, so a malformed flow on document A (e.g. Convert throws before Return) drops the list at the GC and ALSO leaves document B starved of one slot. Impact: bounded (8 lists total), but the cross-document coupling is non-obvious.

## Memory hazards â€” subscription leaks

Audit basis: every `+=` event subscription in `Packages/com.wevaui/Runtime` was matched against its corresponding `-=`. The Document.Mutated subscriber set is the load-bearing case (BindingSet, InvalidationTracker, FormControlsRegistry, CascadeEngine.StyleArray all subscribe; all unsubscribe on Dispose / Detach via the `UIDocument.TearDownPipeline` path at `UIDocument.cs:581-601`).

- [x] **MS1** â€” `EventListeners.map` (`Packages/com.wevaui/Runtime/Events/EventListeners.cs:6`). Stored as `Dictionary<Element, List<EventListenerRegistration>>`. Subscribers: any caller of `EventDispatcher.AddEventListener(Element, ...)`. The dictionary entry is removed ONLY when `RemoveListener` is called with the matching tuple and the per-element list empties (`:27-28`). When an `Element` is removed from the DOM via `Document.Mutated(ChildRemoved)`, the listener entries for that element and its descendants are **NOT** compacted. Impact: every removed element with an attached listener leaks one Dictionary entry + one List + one EventListenerRegistration per listener, plus the strong reference to the Element itself (keying the dictionary). On a controller that AppendChild/RemoveChild-cycles list items with `on-click` attributes, every cycle leaks the prior child's `Element` and handler. **No unsubscribe at:** `Document.Mutated(ChildRemoved)` â€” the dispatcher should listen and call `RemoveListener` for every entry under the removed subtree. Fixed 2026-05-24: EventDispatcher now implements IDisposable and subscribes to Document.Mutated; ChildRemoved walks the entire removed subtree and evicts every listener-bearing Element (plus any dispatcher-internal hovered/focused/pointer-capture/pointerDown references into the subtree); disposed in UIDocument.TearDownPipeline alongside Bindings / FormControls. Regression coverage: `Packages/com.wevaui/Tests/Runtime/Events/EventDispatcherListenerLeakTests.cs` (4 tests).
- [x] **MS2** â€” `CssAnimationRunner` element-keyed state. `Packages/com.wevaui/Runtime/Css/Animation/CssAnimationRunner.cs:16-47, :871`. Six element-keyed dicts/sets â€” `transitions` (`Dictionary<(Element, string), ...>`), `animations` (same), `animationsByElement`, `transitionsByElement`, `animatedElements` (HashSet), `transitioningElements` (HashSet), `elementSpecsCache` (Dictionary<Element, SpecsCache>), `composedCache` (Dictionary<Element, ComposedEntry>) â€” all hold strong Element references. Cleared by `Stop(Element)` and `StopAll()`, but **NO caller invokes `Stop(element)` on DOM removal**. Grep across `Packages/com.wevaui/Runtime` finds zero callers of `Animator.Stop(element)` outside the runner's own code. Active transitions / animations on a removed element keep that element pinned until the animation naturally completes (the per-tick sweep at `:760-772` removes finished records by their `(Element, string)` key); a removed element with an infinite-iteration `animation` (`animation-iteration-count: infinite`) is **pinned forever**. Cross-references AS2 in the existing audit, which flagged the same hazard from the async/coroutine angle. Fixed 2026-05-24: CssAnimationRunner now implements IDisposable and subscribes to Document.Mutated via `AttachToDocument`; ChildRemoved walks the entire removed subtree and routes every Element through a single `RemoveElement` helper that strips it from all **eight** element-keyed dictionaries (the prior `Stop(Element)` path missed `composedCache`, now fixed too); wired in UIDocumentBuilder.Build and HotReloadCoordinator (dispose-old-then-attach-new), and disposed in UIDocument.TearDownPipeline alongside Events. Cancel events are intentionally NOT dispatched per AuthoringGuide Â§16 (H3 out-of-scope). Regression coverage: `Packages/com.wevaui/Tests/Runtime/Css/Animation/CssAnimationRunnerMutationCleanupTests.cs` (5 tests, including the canonical `animation-iteration-count: infinite` leak).
- [x] **MS3** â€” `transformFnCache` keyed by raw string (`Packages/com.wevaui/Runtime/Css/Animation/ValueInterpolator.cs:369`) and `identityListCache` (`:1024`) are process-static and grow up to their cap (256 / 64). When the cap is reached new entries are simply DROPPED (`:523-525`), not evicted â€” the cache effectively "freezes" on the first 256 distinct transform strings. Subsequent novel transforms re-parse on every Tick forever. Not a memory leak (bounded) but a performance cliff â€” flagging because once the cache hits its cap, every novel keyframe permanently pays the parse cost. Fixed 2026-05-24: both caches now use drop-one-on-overflow eviction (arbitrary first-enumerated key removed before the new insert) so the cache stays responsive to the current working set; added `ResetCaches_TestOnly` mirroring the MC1 fix pattern. Regression coverage: `Packages/com.wevaui/Tests/Runtime/Css/Animation/ValueInterpolatorTransformCacheEvictionTests.cs` (5 tests).
- [x] **MS4** â€” Fixed 2026-05-24: BoxToPaintConverter subscribes to a process-global `Box.Recycled` action fired from `Box.ResetForPool` (which `LayoutPools.Recycle` invokes for every retired box); the handler drops the dictionary entry and calls `snapshot.Recycle()` so pooled buffers are reclaimed instead of leaked, AND prevents stale-snapshot replay if the same Box instance is later re-acquired for a different element. Regression coverage: `Packages/com.wevaui/Tests/Runtime/Paint/Conversion/BoxSnapshotRecycleEvictionTests.cs` (4 tests, including the recycled-then-reused regression pin).
- [x] **MS5** â€” `InteractionStateProvider.states` (`Packages/com.wevaui/Runtime/Events/InteractionStateProvider.cs`). `Dictionary<Element, ElementState>` keyed on Element. Entries are added via `SetFlag` whenever an element gains a pseudo-class bit (Hover, FocusWithin, Focus, FocusVisible, Active, Target). Verified 2026-05-24: Hover / FocusWithin self-clean via `DiffApplyFlagChain` (the next chain update strips the flag from every element no longer in the chain â€” works whether or not the element is still in the DOM). Active self-cleans on PointerUp via the dispatcher. But **Focus / FocusVisible / Target do NOT self-clean on DOM removal** â€” the dispatcher's `ForgetIfInSubtree` nulls its own `focused` field, then `FocusInternal` reads `previous = focused` (now null) and skips the SetFlag(previous, Focus, false) cleanup, stranding both bits against the orphaned Element key. Same shape as MS1 / MS2 â€” the provider had no Document.Mutated subscription. Fixed 2026-05-24: `InteractionStateProvider` now implements `IDisposable` and exposes `AttachToDocument(Document)`; ChildRemoved walks the entire removed subtree and evicts every descendant Element from `states` (also nulls `targetElement` if the prior :target was inside the removed subtree, so the next `SetTargetElement` doesn't try to flag-clear an orphan). Wired in `UIDocumentBuilder` step 4 (the provider survives hot reload â€” only Cascade / Animator are replaced), disposed in `UIDocument.TearDownPipeline` alongside Events / Animator. Regression coverage: `Packages/com.wevaui/Tests/Runtime/Events/InteractionStateProviderLeakTests.cs` (5 tests, including the canonical focused-element-removed leak and a multi-flag subtree-removal case covering Focus / FocusVisible / FocusWithin / Target in one pass).

## Memory hazards â€” unbounded static caches

Audit basis: each `static readonly Dictionary<,>` / `HashSet<>` / `List<>` was checked for a soft-cap or compaction. Eight findings, ranked by realistic growth in a long-running session.

- [x] **MC1** â€” `BackgroundResolver.gradientNoCache` (`Packages/com.wevaui/Runtime/Paint/Conversion/BackgroundResolver.cs:476`). `HashSet<string>` of raw gradient strings that failed to parse OR that contain `currentcolor`. Grows on: every novel `currentcolor`-containing gradient string the cascade resolves. **Never compacted.** A controller that animates a gradient's `from`/`to` color while keeping `currentcolor` in the stop list (or just authors many distinct `currentcolor`-bearing gradients) leaks one string entry per distinct text. The sibling `gradientCache` (`:472`) and `gradientBrushCache` (`:431`) DO have soft caps (256 each); only the no-cache set is unbounded. How it might be bounded: a soft-cap clear identical to `gradientCache`'s 256-entry guard. **Fix:** added `GradientNoCacheCap = 256` with drop-new-on-overflow matching the sibling caches, plus `ResetCaches_TestOnly()` for test isolation; regression coverage in `BackgroundResolverGradientNoCacheTests` (2026-05-24).
- [ ] **MC2** â€” `BindingResolver.cache` (`Packages/com.wevaui/Runtime/Binding/BindingResolver.cs:31`). `Dictionary<CacheKey, MemberInfo>` keyed on `(Type, segmentString)`. Grows on: every fresh `(controllerType, bindingPath-segment)` tuple the reflection lookup encounters. Bounded by the application's total `[UIBind]` surface (number of distinct controller types Ă— distinct binding-path segments) â€” effectively static after warmup. **Never compacted.** Cross-references **P20** in the existing audit. Not a real leak for steady-state apps; flag retained because dynamic plugin systems that load new controller types at runtime would grow the cache unboundedly with no mechanism to release entries for unloaded types.
- [x] **MC3** â€” Fixed 2026-05-24: added public `BindingResolver.UnregisterAccessor(Type)` and `UnregisterAccessorFactory(Type)` (both return `bool`, throw on null type, no-op when no entry exists) plus internal `ResetRegistrations_TestOnly()` mirroring `ColorResolver.ResetWarnings_TestOnly()` for bulk test [SetUp]. Old `ClearRegisteredAccessorsForTests()` kept as a thin alias for back-compat. Regression coverage: `Packages/com.wevaui/Tests/Runtime/Binding/BindingResolverRegistrationLifecycleTests.cs` (8 tests: register-then-unregister of both kinds, null-arg guards, unregister-of-unregistered no-op, bulk Reset clears both dicts, Reset idempotent on empty state).
- [ ] **MC4** â€” `TextRunSnapshotCache.cache` (`Packages/com.wevaui/Runtime/Rendering/URP/TextRunSnapshotCache.cs:31`). `Dictionary<TextRunSnapshotKey, TextRunSnapshot>` keyed on (text, font, color, decoration, letter-spacing, blur). Soft-capped at 2048 with **whole-dictionary Clear() on overflow** (`:42`). Functional bound (not a leak) but the Clear-on-overflow policy is non-LRU: a single hotspot text run gets evicted alongside cold entries when the cap trips. Per the class comment at `:21-26` this is by design. Flag retained because the `Clear()` strategy means cache **hit-rate** can drop to zero on a frame where the cap is exceeded, and the subsequent re-population takes hundreds of glyph-bake misses.
- [ ] **MC5** â€” `ColorResolver.s_WarnedKeys` (`Packages/com.wevaui/Runtime/Paint/Conversion/ColorResolver.cs:33`). `HashSet<string>` of "DD2:file:line" or "DD3:rawToken" keys. Grows on: every unique null-style call site (DD2) or every unique malformed color token (DD3). **Never cleared** per the class comment (`:30-31`, "intentionally never cleared â€” 'once per session' matches the pattern of W4/W5"). Bounded in practice by the small set of distinct bad inputs in any one app; **unbounded in principle** if an author generates per-frame dynamic colors that all fail to parse (e.g. a `Random.value`-driven `#XX` color with malformed alpha). Impact: ~50 B per unique bad token, never reclaimed. Documented behaviour â€” flag retained for visibility.
- [ ] **MC6** â€” `AtlasRegistry`'s static dicts (`Packages/com.wevaui/Runtime/Text/Sdf/AtlasRegistry.cs:29-41`). `atlases` / `atlasIds` / `atlasById` / `colorAtlasIds` / `coverageAtlasIds`. The public `UnregisterAtlas` API removes entries from `atlases`, `atlasIds`, `atlasById` but **does NOT remove from `colorAtlasIds` / `coverageAtlasIds`** (verify in source). If the asymmetry holds, the two HashSet<int>s grow per registered atlas and never shrink. **Compaction option:** sync the remove paths.
- [ ] **MC7** â€” `FilterResolver.s_FilterChainCache` (`Packages/com.wevaui/Runtime/Paint/Conversion/FilterResolver.cs:29`). 128-entry soft cap, drop-new-on-overflow (`:60, :79` â€” `if (cache.Count < cap) cache[key] = ...`). Bounded but non-LRU. A scene with >128 distinct (filter-text, currentColor) tuples saturates the cache; subsequent filter resolutions miss forever and pay the FilterParser cost per call. Same flavour as MC4.
- [ ] **MC8** â€” `BoxShadowResolver.s_ShadowCache` (`Packages/com.wevaui/Runtime/Paint/Conversion/BoxShadowResolver.cs:51`). 128-entry soft cap, drop-new-on-overflow. Same shape as MC7. Saturation = permanent re-parse cost on cache-miss tuples.

## Memory hazards â€” undisposed disposables

Audit basis: 14 `IDisposable` types in `Packages/com.wevaui/Runtime`. Each was checked for an owning field and a matching `Dispose()` call. The Wire/Release pair at `InputController` and the `FormControlsRegistry.Dispose()` (which Releases all controllers, `Packages/com.wevaui/Runtime/Forms/FormControlsRegistry.cs:101-110`) cover the bulk of the surface. `UIDocument.TearDownPipeline` at `Packages/com.wevaui/Runtime/UIDocument.cs:581-601` disposes CssWatcher, HtmlWatcher, Bindings, FormControls, ScrollEvents. Two gaps:

- [ ] **MD1** â€” `CssAnimationRunner` is **NOT IDisposable** despite holding element-keyed state across multiple internal Dictionaries (see MS2). `UIDocument.TearDownPipeline` (`UIDocument.cs:581-601`) does not call `state.Animator.StopAll()` either, so on `OnDisable` the runner's per-element references survive until the next BuildPipeline drops the reference to the entire `state`. Functional â€” `state = null` at line 600 releases the runner â€” but a long-running gameplay session that toggles `UIDocument.enabled` repeatedly accumulates animation records across cycles if the same Element instances persist (they don't in normal usage, but a controller that retains DOM nodes across rebuilds would). Suggest implementing `IDisposable` and calling `StopAll()` on `OnDisable`.
- [ ] **MD2** â€” `FormElement` (`Packages/com.wevaui/Runtime/Forms/FormElement.cs:7`) constructs three `Dictionary<Element, ...>` fields (`defaultValues`, `defaultChecked`, `defaultSelected`) populated by `CaptureDefaults()` at construction (`:40-55`). Subscribers: anyone wrapping a `<form>` Element via `new FormElement(formEl)`. **Never disposed** â€” `FormElement` is not IDisposable and the dictionaries hold strong Element references for the original child controls. If a form's children change at runtime (controller-driven add/remove of `<input>` rows), `FormElement.DoReset()` (`:142-167`) reads stale entries for removed children, and the dictionaries pin the removed Elements. Impact: per form, ~24 B per removed control Ă— lifetime of the FormElement instance. Lower-severity than MS1/MS2 because FormElement instances are typically constructed per-submit and discarded, but the API contract isn't clear about that.
- [ ] **MD3** â€” `URPRenderBackend.filterPipeline` is correctly disposed at `Packages/com.wevaui/Runtime/Rendering/Backend/URPRenderBackend.cs:508`. Not a finding â€” flagged here as a positive control to confirm the audit caught the dispose call. (Same for `BindingSet`, `FormControlsRegistry`, `InputController`, `ScrollEventHandler`, `CssWatcher`, `HtmlWatcher` â€” all correctly disposed via `UIDocument.TearDownPipeline`.)

## Memory hazards â€” captured-in-closure references

Audit basis: every `Action<>` / `delegate` field that captures `this` in a long-lived disposable.

- [ ] **MX1** â€” `BindingSet.mutationListener` (`Packages/com.wevaui/Runtime/Binding/BindingSet.cs:26, :86`). Captures the BindingSet instance (`mutationListener = OnMutation`); subscribed to `Document.Mutated`. Correctly detached at `DetachLive()` (`:90-97`) and `Dispose()` (`:174`). No leak â€” positive control.
- [ ] **MX2** â€” `FormControlsRegistry.mutationListener` (`Packages/com.wevaui/Runtime/Forms/FormControlsRegistry.cs:24, :31`). Same pattern as MX1. Correctly removed in `Dispose()` (`:104`). No leak â€” positive control.
- [ ] **MX3** â€” `InvalidationTracker.OnMutation` (`Packages/com.wevaui/Runtime/Reactive/InvalidationTracker.cs:235`) is a method group conversion. Subscribed via `doc.Mutated += OnMutation` and detached via `doc.Mutated -= OnMutation` (`:241`). The method-group conversion **allocates a fresh delegate on each `+=` AND `-=` call**, so the `-=` may not match the `+=` delegate by reference in some Mono/IL2CPP configurations â€” though .NET semantics specify open-delegate equality should hold. Worth verifying: if `+=` allocated delegate A and `-=` allocates delegate B (different references but same Target+Method), Delegate.Equals returns true, so the unsubscribe succeeds. Flag retained as a known footgun for future maintainers.
