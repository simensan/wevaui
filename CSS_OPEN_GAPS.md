## A. Runtime bugs / spec divergence — REAL behaviour differs from spec

### ~~A-SRGB-COMPOSITE. UI layers composited in linear light, not Chrome's sRGB gamma space~~
- **CLOSED — gamma-space compositing is now the default** (2026-06-09, live-verified on `glass.html` in the editor game view across every stage). Severity was `runtime`. Spec: CSS Color 4 §2 / Compositing 1 — CSS alpha compositing of stacked layers happens in **gamma sRGB**, not linear-light. The engine stored every CSS color as true-linear (`LinearColor.FromCssColor` → `SrgbByteToLinear`) and blended premultiplied `One/OneMinusSrcAlpha` into the **linear** camera target, so a bright color over a dark background lifted the green/luma channel far more than Chrome.
- **Reported case**: `glass.html` backdrop blobs — teal `#5eead4`@0.7 / blue radial fades over the navy `linear-gradient(#0b0a1e→#161236→#0d0b24)` base flooded teal/green vs Chrome's deep navy. (Hand math: gamma-space over ≈ (70,167,159); linear-space over ≈ (78,201,183) — green 167→201.)
- **Fix**: the UI now renders into an always-on intermediate UNORM (sRGB=false) RT holding **sRGB-encoded premultiplied** values, so the FF blend composites in gamma space (Chrome-exact over ANY background); a final pass decodes sRGB-premul→linear-premul into the linear camera (Linear projects) or passes through (Gamma). Eliminates the old `Weva_CssLinearTargetBlendLift = 0.16` approximation (which was only exact against black). Gated on `ShaderResources.UseSrgbComposite` (now default **true**) via the per-frame `_WevaSrgbComposite` global; the legacy 0.16 path + mode-17 stay as the flag-off A/B fallback (keeps `StencilWriteShaderContractTests` green).
- **Stages, all committed** 51beb705..05c9cec2: 0 encode seam + flag; 1 intermediate RT + composite-back (orientation derived from `isActiveTargetBackBuffer`, backdrop-filter capture flips because the live target is now an RT); 2 exact sRGB-premul encode + sRGB=false intermediate + decode-on-composite; 3 mode-17 falls through (FF blend is exact sRGB source-over) + mix-blend dst treated as pre-encoded; 4 backdrop-filter decodes sRGB→linear before the blur chain and re-encodes on composite-back; 5 `BatchedSurfaceRenderer` sets `_WevaSrgbComposite` deterministically; 6 default-on + dead `_WEVA_SRGB_COMPOSITE` keyword removed.
- **Caveats / follow-ups**: (a) GPU goldens (`GameUIGpuGoldenTests`, snippets 29-38) are gated `#if WEVA_URP` but the Tests asmdef only defines `WEVA_URP_BATCHER_TESTS` — they're compiled OUT and never run, so their `Baselines.GPU/` PNGs are stale-but-unread; regenerate when that gate is fixed (same define-mismatch family as the repeating-gradient test). Software goldens (Golden_10/17, SoftwareRasterizer) are a separate path, unaffected. (b) Minor residual deltas vs Chrome remain (user-accepted); the blob "different spot" was animation phase, not colour. (c) Consumers (a real game UI) inherit the new default — validate the non-backbuffer composite/backdrop-filter flip there (only the backbuffer case was live-validated; `SrgbCompositeSourceYFlipOverride` is the debug escape hatch). (d) `plus-lighter` mix-blend stays linear v1.

### ~~ATPROP-2. `@property` rule with `initial-value: var(...)` incorrectly accepted~~

### A-ABSPOS-PIN-CHILD-CB. Block children of an inset-pinned abspos keep the PROVISIONAL containing-block width
- **Severity**: runtime
- **Symptom**: an absolutely positioned box with `left` + `right` both set and `width: auto` resolves its own width correctly per CSS2 §10.3.7 (CB width − insets − margins), but its in-flow **block** children stay at the width they were given during the provisional BlockLayout pass (the CB's full content width). Minimal repro (450 px viewport): `.abs { position:absolute; left:16px; right:16px }` → `.abs` = 418 px, but a plain `div` child = **450 px** (Chrome: 418 px), overflowing the parent's right edge by 32 px.
- **Found via**: `neon.html` portrait HUD sample (2026-06-12) — `.hud-bottom { left:12px; right:12px }` meter rows ran past the panel edge; confirmed with the BaselineGen `layout-dump` repro above.
- **Suspected location**: `PositioningPass.ApplyAbsoluteAgainst`, horizPinned branch (~line 410). The branch ALREADY calls `block.RelayoutContentAt(pinnedBlock, w)` when the pinned width changes (the nook-dialogue fix, CSS2 §10.3.7 comment) — but in the repro the child still reports the provisional width, so either the relayout isn't re-applying the box model to block children's widths, the `pinOnly` second sweep re-stomps them, or the repro path never enters the non-pinOnly call. Needs a debug trace before touching the relayout contract (see the PAINT-1 closure for how fragile this area is).
- **Workaround**: make the pinned abspos a flex/grid container (`display:flex; flex-direction:column`) — those re-run their own pass at the pinned width after RepositionAbsolutes and size children correctly. `neon.css` `.hud-bottom` uses this and documents it.
- **Spec §**: CSS2 §10.3.7 (abspos width resolution) + §10.1 (containing block for in-flow descendants = the abspos box's content box).
- **Pinned by**: no test yet — add a `[Ignore]`-d repro (parent w=418 / child w=418 at 450 vw) when picked up.
- **Opened**: 2026-06-12.

### ~~A6. Backdrop-filter sample-region leak — root cause: stale pre-UI snapshot~~
- **CLOSED 2026-06-01** — root-caused in `UIRenderGraphPass.ExecuteBatched`:
  the backdrop copy RT was populated ONCE by `TryBindBackdropCopy` (a
  single Blit of the camera color target into a temp RT BEFORE any UI
  batches drained). Per CSS Filter Effects L2 §3, the backdrop is "the
  contents of the canvas... directly behind the element... including
  any other content that has been rendered prior to this element."
  That MUST include UI batches painted earlier in the SAME frame. The
  single-shot copy captured only the pre-UI framebuffer (3D scene +
  post-FX), so any UI painted earlier (body bg, dashboard root,
  sibling panels) never appeared in the sample — `backdrop-filter`
  blurs sampled stale framebuffer content (Editor scene gizmos,
  previous-frame remnants), visible as ghost smudges in any sticky
  header or translucent overlay.
- **Fix**: refresh the backdrop copy from the current `colorTarget`
  immediately before each `ApplyBackdropAndComposite` call in
  `DrainBatches`. `cmd.Blit(colorTarget, backdropSourceTarget)` + RT
  restore. Two call sites: per-batch and drain-tail. Skipped when
  `backdropSourceTarget.Equals(colorTarget)` (no-copy fallback).
- **Visually verified**: the reference game's `.discovery-toast` and the advanced
  dashboard `.hdr` backdrop-filter both now blur the actual content
  underneath (dashboard bg, scrolled-under cards) instead of pre-UI
  stale framebuffer.
- **Tradeoff**: one extra screen-sized blit per backdrop-filter event
  per frame. For pages with many translucent overlays this could add
  up; a future optimisation is to skip the refresh when no UI batches
  have been issued since the previous copy.

### ~~FILTER-SCOPE-GHOST. Filtered element composited Y-mirrored (flipped drop-shadow / text-shadow ghost)~~
- **CLOSED 2026-06-03** (commits `44b64648` + `dfb59570`). Two independent Y-flip bugs in the URP filter compositor, same family, both surfaced by `story-bubble.html` (`.frame { filter: drop-shadow(0 16px 24px …) }` + `.line { text-shadow: 0 2px 3px … }`):
  1. **Drop-shadow / large blur** (`44b64648`): `ApplyBlur`'s downsample pass (radius ≥12px → ×2, ≥32px → ×4) blitted with `sourceYFlip: InternalFilterSourceYFlip`, toggling orientation a 3rd time on top of the self-cancelling H+V passes. The surrounding code (and `ApplyDropShadow`'s tint-in-lockstep-with-source) assumes blur is orientation-neutral, so downsampling blurs came out mirrored — `blur(24)` drop-shadow rendered on the wrong side. Small blurs (menu card-shadow, ×1) never downsampled → unaffected. Fix: downsample blit `sourceYFlip: false`.
  2. **Blur-final scope / small blur** (`dfb59570`): when a scope's final filter is a blur, the last V-pass writes straight into the persistent cache RT in the flipped orientation, but `EndScopeAndComposite` composited it via `CompositeCachedScope` with a hardcoded `sourceYFlip=0` → blurred result Y-mirrored. A `text-shadow` (blur scope) put its shadow ABOVE the glyphs. The CopyTexture path (drop-shadow / color-matrix finals) needs no flip, so the correct composite flip is per-scope: store `FilterScopeCache.CompositeYFlip` (blur-final → flip, CopyTexture → none), consumed by `CompositeCachedScope` for both fresh and cached-replay paths.
- **Verified** via a deterministic offscreen capture (force-build layout + `camera.Render()` to a RenderTexture + `ReadPixels` — see note): story-bubble with both filters on renders the drop-shadow below the panel, no ghost, text-shadow correct. Filter/render-pass suites 46/46.
- **Tooling note**: in-editor `ScreenCapture.CaptureScreenshot*` was unusable this session (game view doesn't repaint while unfocused → stale/null/flipped frames; the play-mode scene reverts the document). The reliable path that finally worked: set `DocumentAsset`, `PrepareForRenderViewport(w,h)`, `UIDocumentLifecycle.RunLayout(state)` (the live deferred rebuild needs the play loop, which doesn't advance from script), then `camera.Render()` to an ARGB32 RT and `ReadPixels`. Do ONE capture per fresh build — repeated `DocumentAsset`/`RunLayout` on a reused state corrupts layout (spurious letter-spacing).
- **Note**: `InvalidateScopeCaches` wiring (`0134e009`) was an unrelated latent fix discovered en route, not this bug.

### ~~FLEX-MINHEIGHT-FILL. `flex: 1` children don't fill a `min-height`-floored flex container~~ (FIXED 2026-06-04)
- **Severity**: runtime (flex sizing). Spec: CSS Flexbox §9.7 + Box Sizing L3 §5.
- **Symptom**: a column flex container sized by `min-height` (taller than its content — e.g. a page root `min-height: 100vh`) does NOT distribute the floor's free space to `flex-grow` children. A `flex: 1` child (and a `grid-auto-rows: 1fr` grid inside it) stays at content height instead of stretching to fill, leaving empty space at the bottom. Chrome stretches them. Canonical: gold-shop's `.packs { flex: 1; display: grid; grid-auto-rows: 1fr }` inside `.shop { min-height: 100vh; display: flex; flex-direction: column }` — the pack cards don't extend to fill the viewport.
- **Root cause (multi-layer circularity — fully traced 2026-06-04)**: this is NOT a single missing definiteness check; it is a feedback loop across three layout algorithms that the current pipeline cannot resolve without a stretch-independent intrinsic-block-size pass. The pieces, each of which was individually fixed during the trace yet still balloons in combination:
  1. `FlexLayout.HasDefiniteMain` ignores `min-height` (only explicit `height`/grid-stretch/abs-pin count), so grow distribution is skipped. (`HasDefiniteMainSize`, the button-`justify` default, DOES honour min-height — diverged.) *Fixable.*

  3. The fatal loop: the grid's `grid-auto-rows:1fr` rows fill whatever Height the grid currently has, so the grid's *content height* equals its *assigned height* (circular). The flex base size of a column grid item is `box.Height` (no intrinsic-block-size measurement exists — only `MaxContentWidth` for the row axis), so the FILLED height feeds back into the flex base. Worse, the grid's row-track sizing reads each card's `IntrinsicCross` which for a stretched flex card is its FILLED height too — so even an indefinite-height probe measures filled content. And `BlockLayout` independently sizes the flex container's Height by stacking its children's *filled* heights, so even after `FlexLayout.FinalizeContainerMainSize` correctly recomputes `.shop`=917 from base sizes, BlockLayout re-inflates it to 2601 from the filled packs. Every input that should be "intrinsic content" is actually "current filled size", so the system has a stable fixed point at the ballooned size (shop 917→2601, packs→2459, cards→1219).
- **Fix path (CAREFUL — every localized attempt ballooned, 2026-06-04)**: a `GridBox.IntrinsicContentBlockSize` stamp (row tracks resolved against ZERO available space, gated to flex-column grid items) computes the correct stretch-independent height (646) and feeding it into `ComputeBaseSize` worked on most sub-passes — but a transient `ICBS=0` sub-pass (grid not yet laid out that frame) falls back to the ballooned `box.Height` and starts an unrecoverable balloon, AND BlockLayout's child-stacking sustains the balloon regardless. A correct fix needs a genuine stretch-independent intrinsic-block-size resolution used consistently by (a) flex base sizing, (b) grid row-track item measurement, and (c) BlockLayout's sizing of flex containers — i.e. content heights must be measured with `align`/`1fr` stretch DISABLED, independent of any flex/grid-assigned size. That is an architectural addition (vertical intrinsic sizing), not a localized patch. Reverted to keep layout stable (shop 917, packs 646, cards 309 — cards fill the grid; only the bottom ~130px viewport gap remains).
- **Resolution (2026-06-04)**: introduced a non-destructive stretch-free intrinsic block-size, `GridLayout.MaxContentBlockSize(grid)` — it runs the full grid sizing prologue in a `measureOnly` mode that forces the row axis indefinite (`1fr`-as-content) and returns `rowsTotal + frame` before positioning any item, so it depends on no flex/grid-assigned Height. Three consumers now use it for a column-flex grid item: (a) `FlexLayout.ComputeBaseSize` (the flex base size), and (b) `FlexLayout.AutomaticMinimumMainSize` (the `min-height:auto` floor, which previously used `GridIntrinsicCross` = the FILLED child extents). With both the base AND the auto-minimum stretch-free, `FinalizeContainerMainSize` (now summing the HYPOTHETICAL/pre-grow item sizes) settles `.shop` at its min-height; flex-grow then distributes the floored space (`HasDefiniteMain` + `containerMainSize` now honour min-height), and the grid fills the flex-assigned Height (`IsFlexParentDefiniteMainColumn` + `FinalizeContainerSize` honour min-height). Result on gold-shop (viewport 1738×917): `shop=917, packs=775, cards=377` — fills the viewport like Chrome, stable across all passes, no balloon. Verified by a live render. 413 Flex+Grid and 1764 Layout tests green; coverage in `FlexMinHeightGridFillTests`.
