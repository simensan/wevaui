**This is NOT a ready-to-fix backlog.** Findings are from automated agent inspection — many have not been independently confirmed. Before fixing any item, require **one** of:
- A failing NUnit repro that pins the current wrong behaviour, OR
- A side-by-side Chrome comparison fixture demonstrating the divergence.

A second-pass reviewer (2026-05-22) confirmed the following as matching current code well enough to act on once a repro exists: **A2, A3, A9, A10, A12, A13, A14, A18, A19, A20, B1, D1, J1, J2, J3**. Everything else carries `[ ]` (unverified) or `[?]` (needs verification) and should be treated as a hypothesis until proven.

## How to use this file

1. **Verify first.** Write either a failing NUnit test or a Chrome-comparison fixture that demonstrates the wrong behaviour. If the test passes (engine already correct) or the divergence isn't reproducible, mark the item `[-]` with a one-line note and move on.
2. Once a repro exists, flip `[ ]` → `[~]` and start the fix. The repro becomes the regression test.
3. Mark `[x]` when fixed. Add `→ commit-sha` or `→ PR #N` inline.
4. If the agent's claim turns out partially wrong (e.g. wrong file:line, narrower scope, contradicted by an existing test), mark `[?]` and add the counter-evidence inline — don't silently delete; future audits should know what was checked.
5. Doc-mismatch fixes (Section J) usually land in `PLAN.md`/`CONFORMANCE.md`/`CSS_FEATURE_AUDIT.md`, not code. Confirm impl matches spec before changing docs.
6. When the active vs legacy renderer matters (A4-A6 etc.), say which path was verified.
7. Re-run a fresh audit periodically to surface drift; replace this file rather than letting it grow stale.
